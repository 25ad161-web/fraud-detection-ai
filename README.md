# AI-Enabled Fraud Detection in Digital Banking

An end-to-end machine learning system that detects fraudulent transactions in
digital banking, UPI, and online payment systems in real time. Built with
Python, Scikit-learn, XGBoost, Flask, and a custom dashboard frontend.

## 1. What this project does

Traditional rule-based fraud systems ("flag any transaction over ₹50,000")
miss sophisticated fraud patterns and generate excessive false alarms. This
project instead trains and compares **five machine learning approaches** on
the real-world **Kaggle Credit Card Fraud Detection dataset**, picks the
best performer, and serves it behind a REST API with a live dashboard.

**Models trained and compared:**
| Model | Type |
|---|---|
| Isolation Forest | Unsupervised anomaly detection |
| Local Outlier Factor | Unsupervised anomaly detection |
| Logistic Regression | Supervised classification |
| Random Forest | Supervised classification |
| XGBoost | Supervised classification |

**Evaluated on:** Accuracy, Precision, Recall, F1-score, ROC-AUC, Confusion Matrix.

**Includes:**
- Full EDA with saved charts (class imbalance, amount distributions, correlations)
- SMOTE-based class imbalance handling
- A Flask REST API (`/api/predict`, `/api/predict/batch`, `/api/explain`, `/api/stats`, `/api/history`, `/api/model-info`)
- SHAP-based model explainability (why was this transaction flagged?)
- A dark, data-dense dashboard UI: transaction form, live risk gauge, charts, fraud history table
- Logging, input validation, and graceful error handling throughout

## 3. Setup instructions

### 3.1 Prerequisites
- Python 3.10+
- ~500 MB free disk space (dataset + model artifacts)

### 3.2 Clone and install dependencies

```bash
cd fraud-detection
python -m venv venv

# Activate the virtual environment
source venv/bin/activate          # macOS/Linux
venv\Scripts\activate             # Windows

pip install -r requirements.txt
```

### 3.3 Get the dataset

Download the **Credit Card Fraud Detection** dataset from Kaggle:
`https://www.kaggle.com/datasets/mlg-ulb/creditcardfraud`

Place the downloaded `creditcard.csv` into `data/raw/creditcard.csv`.

**Don't have a Kaggle account / want to test the pipeline immediately?**
Run the included generator to create a schema-matching synthetic dataset:

```bash
python scripts/generate_sample_data.py
```

This creates `data/raw/creditcard.csv` with the same columns (`Time, V1-V28,
Amount, Class`) and a realistic ~0.17% fraud rate, so the rest of the
pipeline can be run and tested without Kaggle access. It will NOT overwrite
a real dataset already placed there.

### 3.4 Run the EDA + training pipeline

```bash
python notebooks/01_eda_and_training.py
```

This will:
1. Load and explore the dataset (saves charts to `backend/static/charts/`)
2. Clean missing values and engineer features
3. Split into train/test sets (stratified, 80/20)
4. Scale features and apply SMOTE to the training set only
5. Train all 5 models and evaluate each on the untouched test set
6. Save the best model (`models/best_model.pkl`), the fitted scaler
   (`models/scaler.pkl`), feature column order, and metrics to `models/`

Expect this to take anywhere from under a minute (small/synthetic dataset)
to several minutes (full 284,807-row dataset, mainly for Random Forest/XGBoost).

### 3.5 Start the backend + dashboard

```bash
python backend/app.py
```

Then open **http://localhost:5000** in your browser. The same Flask server
serves both the REST API and the dashboard UI — no separate frontend server
needed.

### 3.6 Run automated tests

```bash
pytest tests/test_api.py -v
```

(Requires the model to already be trained, since the tests call `/api/predict`.)

## 4. Using the dashboard

1. **Transaction Input** — enter an amount (required) and optionally the
   time-of-day in seconds; click "+ Advanced" to enter raw V1-V28 PCA
   features if you have real anonymized data. Click **Analyze Transaction**.
2. **Prediction Result** — a risk gauge shows the model's fraud probability
   (0–1), a verdict banner, the risk tier (Low/Medium/High), and which model
   produced the result. Click **Explain this prediction (SHAP)** to see which
   features pushed the score up or down.
3. **Dashboard cards** — running totals of scanned, flagged, and legitimate
   transactions, and the live fraud rate.
4. **Model Performance & EDA Charts** — tabs to flip through the saved EDA
   and evaluation visualizations generated during training.
5. **Fraud History** — a live table of recent predictions, refreshable on demand.

## 5. REST API reference

| Method | Endpoint | Description |
|---|---|---|
| `POST` | `/api/predict` | Predict fraud for one transaction |
| `POST` | `/api/predict/batch` | Predict fraud for up to 500 transactions |
| `POST` | `/api/explain` | SHAP explanation for one transaction |
| `GET`  | `/api/stats` | Aggregate dashboard stats |
| `GET`  | `/api/history?limit=N` | Recent prediction history |
| `GET`  | `/api/model-info` | Deployed model metadata + comparison table |
| `GET`  | `/api/charts` | List available chart filenames |
| `GET`  | `/health` | Health check (model load status) |

**Example request:**
```bash
curl -X POST http://localhost:5000/api/predict \
  -H "Content-Type: application/json" \
  -d '{"amount": 1899.00, "time": 5000}'
```

**Example response:**
```json
{
  "is_fraud": false,
  "risk_score": 0.4641,
  "risk_label": "Medium",
  "model_used": "XGBoost"
}
```

## 6. Design notes & rationale

- **Why F1-score for model selection?** With ~0.17% fraud, a model that
  always predicts "legitimate" gets ~99.8% accuracy while being completely
  useless. F1-score balances precision (don't cry wolf) and recall (don't
  miss real fraud), which is the actual operational tradeoff a bank cares about.
- **Why SMOTE only on the training set?** Applying SMOTE before the
  train/test split would leak synthetic minority-class patterns into the
  test set, inflating evaluation metrics in a way that wouldn't hold up on
  real, naturally-imbalanced incoming transactions.
- **Why a shared `preprocessing.py`?** The exact same feature engineering
  code path is used at training time and at API inference time, preventing
  "training/serving skew" — a common, hard-to-debug class of real-world ML bugs.
- **Why JSON-file history instead of a database?** Keeps the project
  dependency-free and the data trivially inspectable for grading/demo
  purposes, while still being real persistence across server restarts.

## 7. Extending this project

- Swap the JSON history store for PostgreSQL/SQLite for true concurrent-write safety.
- Add authentication (API keys / JWT) before exposing `/api/predict` publicly.
- Containerize with Docker and front with a production WSGI server (gunicorn) + nginx.
- Add a model retraining schedule / drift monitoring as transaction patterns evolve.
- Replace the gauge gimmick, if desired, with a full transaction-stream simulator for live demos.
